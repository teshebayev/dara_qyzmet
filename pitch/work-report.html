🎯 Постановка задачи

Два направления, которые мы взяли в работу

Изначально это были две отдельные темы. Мы реализовали обе и объединили в один сквозной поток: распознали накладную → приняли товар → зафиксировали расхождения → сформировали акт.

🔡
Smart Invoice Scan — OCR/VLM

Боль: магазины вручную проверяют товар, повторно вводят данные, ставят цены.

Задача: сфотографировал накладную → AI распознал поставщика, номер, дату, список товаров, количество, цены → позиции автоматически попадают в учётную систему.

≥80% точность полей≥5 полей редактирование до подтверждения

⚖️
Smart Acceptance — цифровая приёмка

Боль: приём по бумажным накладным, сверка вручную, расхождения теряются.

Задача: дистрибьютор отправляет электронную накладную → магазин подтверждает / вычёркивает / корректирует / фиксирует брак с фото → система формирует акт расхождений → дистрибьютор корректирует счёт.

≥2 сценария расхожденийавтокорректировка суммы роль дистрибьютора

🔗 Как объединили

Один бэкенд и одна модель данных обслуживают оба направления: распознанная накладная становится входом для приёмки, а расхождения пересчитывают итоговую сумму «к оплате» в едином месте — это убрало дублирование и дало сквозной сценарий «фото → акт».

⚖️ Направление 2 · Smart Acceptance

Цифровая приёмка с расхождениями и актом

🔄 Машина состояний заявки

new → shipped → receiving →
{ accepted | discrepancy } →
act_created → invoice_corrected → closed

Переходы валидируются (assert_transition) — нельзя «перескочить» статус. Принятые товары уходят в сток только после подтверждения приёмки.

⚖️ 4 типа расхождений → автопересчёт

✓Недостача / излишек — изменение факта количества
✓Пересорт — пришёл не тот товар
✓Брак — обязательно с photo_url (фотофиксация)
✓Сумма «к оплате» пересчитывается в recalc.py: Decimal + ROUND_HALF_EVEN

📑 Акт расхождений

При создании акта фиксируется снимок: исходная сумма / скорректированная / дельта. Заявка переходит в act_created, и поставщик получает акт на корректировку счёта.

👥 Роль дистрибьютора (поставщика)

Отдельный кабинет: поставщик видит акты расхождений и корректирует счёт. Заявка переходит в invoice_corrected. Так решение не «однобокое со стороны магазина» — оба контрагента в одном процессе.

Smart Acceptance Приёмка · ≥2 сценария, автокорректировка суммы, акт
📱 Мобильное приложение

Терминал сбора данных (ТСД) для приёмки

Отдельное мобильное приложение halyk-dct превращает телефон в терминал сбора данных: кладовщик в зале приёмки сканирует позиции и фиксирует расхождения, не возвращаясь к компьютеру. Данные уходят в тот же бэкенд, что и web/бот.

Роль в продукте
✓Сканирование товара при приёмке — сверка факта с накладной прямо на складе.
✓Фотофиксация брака — снимок прикрепляется к расхождению (тип defect).
✓Мобильность — приёмка идёт у стеллажа, а не «на бумаге, потом в Excel».
✓Единый API — приложение ходит в тот же REST-бэкенд (/api/v1).

🤖 AI-агент

Function-calling поверх данных склада

LLM сама выбирает инструмент (через узкую guided_json-схему решения), получает результат из БД и свободно формулирует ответ. Все инструменты — read-only и фильтруются по org_id.

Примеры: «сколько молока на складе?», «какие расхождения?», «сколько потратили у Молпрома за май?», «что на исходе и у кого дозаказать?».

🛡️ Ключевая инвариант безопасности: «предложи → подтверди»

✓LLM не пишет в БД. У неё есть только propose_order_draft (read-only).
✓Черновик заказа создаёт детерминированный узел — только после подтверждения человеком.
✓Деньги модель не считает — за суммы отвечает recalc.py.

📈 Наблюдаемость

Все вызовы (решение, инструменты, финальный ответ, токены) трассируются через Langfuse — видно, какой инструмент выбрала модель и с какими аргументами. Это и помогло найти проблему, о которой на следующем слайде.

Известные ограничения и план доработок

⚠️
Главное: ВЛМ-агент иногда генерирует неправильные параметры для тула

Слабая/локальная модель при function-calling нередко выдумывает имена аргументов и лишние ключи (например, name вместо query, vendor вместо supplier), сужает type расхождения до отсутствующего значения и получает пустой результат, после чего отвечает «расхождений нет».

Как уже подстраховались в коде
✓Алиасы аргументов (_ARG_ALIASES): синонимы приводятся к ожидаемым именам (name→query, vendor→supplier…).
✓Фильтрация лишних ключей (_prepare_args): выдуманные параметры отбрасываются, чтобы не ронять тул (TypeError).
✓guided_json на этапе выбора инструмента — узкая схема решения.
✓Обход сужения type: discrepancy_report всегда отдаёт полную разбивку по типам.
✓Фолбэк по фото: если модель не вызвала поиск по фото — делаем это сами.
Что доработать дальше
›JSON-schema на аргументы каждого тула через Pydantic (а не только на решение): модель аргументов на pydantic → авто-генерация схемы и строгая валидация перед вызовом.
›Авто-ретрай: при невалидных аргументах возвращать модели ошибку схемы и просить исправить.
›Более сильная модель для tool-calling или нативный tools API вместо ручного парсинга JSON.
›Метрики качества вызовов из Langfuse: доля невалидных аргументов, пустых ответов.
›Few-shot примеры корректных вызовов в системном промпте.
